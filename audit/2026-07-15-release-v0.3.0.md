# Release audit — v0.3.0 (2026-07-15)

Full-feature audit and validation gate for the **v0.3.0** release (the first
tagged release after v0.2.0, 2026-06-21). Verdict: **GO**, conditional on the
two human-only release steps (signed tag + PyPI environment approval) noted in
§6. All automated gates are green; the findings below were doc/code drift and
CHANGELOG completeness, all remediated in this same change.

## 1. Scope & method

Audited the full v0.3.0 surface: the 22-tool MCP contract, both transports,
auth, audit + hash chain, tiered policy + redaction, the Tier-3 confirmation
broker, the seccomp-notify channel, the SSH pool, sessions, config, and metrics
— plus release mechanics (versioning, CHANGELOG, PyPI metadata, the signed-tag
release workflow) and documentation-vs-code consistency (ADR-0005 style).

Method: a baseline gate + scanner battery run locally, and three parallel
read-only deep-dive validations (feature/wiring inventory; docs-vs-code drift;
CHANGELOG + release-readiness). Findings were cross-checked between the passes.

## 5. Findings & dispositions

All findings were doc/code drift or completeness — no functional or security
defect. All are remediated in this change.

**Fixed here:**

1. **Metrics `outcome` label drift** (`docs/deployment.md` §9a) — the documented
   label set (`ok|denied|error`) omitted `confirm_required`, which the Tier-3
   broker emits (`server.py`, `metrics.py`). A SIEM/dashboard operator would not
   have known to expect it. Table corrected.
2. **CHANGELOG completeness** — four user-facing changes since v0.2.0 had no
   `[Unreleased]` entry: the pinned-dependency bump (`mcp` 1.27.2→1.28.1,
   `asyncssh` 2.23.1→2.24.0) + its DOC-1 doc reconciliation, the MkDocs
   documentation site, and the DOC-6 host-integrity monitoring guidance. Added.
3. **Settings reference gap** (`docs/deployment.md`) — the HTTP-listener bind
   (`RELAY_SHELL_HTTP_HOST`/`_HTTP_PORT`) and several session/timeout/limit
   bounds were documented only in `.env.example` / `runbook.md`, not in the
   deployment guide. Added §8c (a pointer to `.env.example` as the complete list
   plus the operator-relevant knobs) and clarified the configurable bind in §4.
4. **PyPI metadata** — added `Documentation` and `Changelog` project URLs.

**Non-blocking follow-ups (tracked, not release blockers):**

- **Auth wiring test.** The `build_server()` HTTP-only + `auth_enabled` gate has
  no direct test; only the `auth/oauth.py` primitives are unit-tested. Low risk
  (a two-clause `if`), but worth a `build_server`-level assertion for parity
  with how the broker/seccomp wiring is covered. → backlog.
- **Streamable-HTTP e2e.** No dedicated MCP-over-HTTP initialize/list/call
  round-trip analogous to `test_stdio_e2e.py`; HTTP correctness is currently
  inferred from route-presence + OAuth tests. → backlog.
- **Docs build on Python 3.12.** `mkdocs build --strict` crashes on 3.12 via a
  stdlib `html.parser` quirk on `(?<![\w])` regex snippets in the runbook /
  ADR 0011; CI builds docs on 3.14 (green) and the Pages site deploys correctly.
  Non-blocking; optionally fence the snippets for cross-version local builds.

## 6. Release mechanics — prepared

Done in this change (agent-doable):

- Version bumped `0.2.0` → `0.3.0` in `pyproject.toml` and
  `src/relay_shell/__init__.py` (both; `server_info` surfaces `__version__`).
- CHANGELOG `[Unreleased]` finalized to `[0.3.0] - 2026-07-15` with a fresh
  empty `[Unreleased]` above it and the `[0.3.0]` compare link.

Remaining — **human-only**, per `.github/workflows/release.yml`:

1. **Create a signed, annotated tag `v0.3.0`** on the release commit
   (`git tag -s v0.3.0 -m "…"`). The `verify` job hard-fails unless GitHub
   reports the tag object as `verification.verified == true` — an agent without
   a signing key cannot produce this.
2. **Push the tag** (`git push origin v0.3.0`) → triggers `release.yml`
   (verify → build → publish). `verify` re-checks the tag equals the tagged
   commit's `pyproject` version (0.3.0 — will pass).
3. **Approve the `pypi` environment** if the repo's environment protection
   requires a reviewer click before `publish` uploads via OIDC trusted
   publishing.
